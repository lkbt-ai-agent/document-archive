# 작업 큐와 작업 상태 TODO

## 범위

긴 업로드 처리, OCR, 임베딩, AI 생성을 서버 프로세스 수명과 분리하고 상태 UX를 개선합니다.

## 백엔드

- 업로드와 AI 생성은 FastAPI `BackgroundTasks`로 분리됐지만 별도 큐/워커는 없습니다.
- 재시도, 진행률, 작업 취소, 장기 작업 안정성을 위한 큐가 필요합니다.
- 작업 ID, 단계, 시작/종료 시각, 실패 사유를 저장할 모델이 필요합니다.

## 프론트엔드

- 현재는 `processing` 문서 반환 후 3초 폴링으로 완료를 확인합니다.
- 단계별 진행률, 취소, 재시도 UX가 없습니다.
- 장기 작업이 여러 개일 때 작업 목록과 상태를 보여줄 수 있어야 합니다.
